# v3.1 — FineWeb base upgrade + SFT distillation

**Parent doc:** [../PLAN.md](../PLAN.md) §7 v3 (extension)

**Builds on:** [`v3-tiny-llm.md`](v3-tiny-llm.md) (MVP closed; this plan re-opens v3
to address the quality gap surfaced by §8 narrative findings)

**Prerequisite:** v3 MVP commits `b839775` … `dc51663` are merged; the §8
experiment narrative documents the "doc-shaped noise" output that motivates
this follow-up

**Estimated duration:** ~3 calendar days (1 day code + 1 overnight pretrain +
1 day SFT + writeup)

**Core goal:** Take the closed v3 MVP TinyDocs from "doc-shaped noise" to
"tries to answer English queries" by

1. Pretraining on a FineWeb-Edu (95 %) + Python-docs (5 %) mix instead of
   docs-only (which forced the model to memorize a 2.6 M-token corpus).
2. SFT-distilling from Qwen2.5-1.5B on grounded RAG queries, scaled to
   1000–2000 samples (vs a 122-sample diagnostic done in the post-MVP
   feasibility exploration).

**Stays a learning-oriented side track**, just like v3. Quality ceiling is
still bounded by the 57M parameter budget; this plan does not chase
Qwen-level quality.

---

## Why this plan exists

Two empirical findings from the v3 MVP and a post-MVP feasibility
exploration (executed on a throwaway branch; raw artifacts not committed):

1. **SFT alone on the docs-only base produced flat results.** A 122-sample
   SFT corpus distilled from Qwen, fine-tuned for 90 steps on the
   `run-10k-b8` ckpt, gave 26 % qualitative wins, 18 % mode-collapse
   regressions, and 56 % no-change outcomes. The diagnostic concluded the
   base LM was the bottleneck, not the SFT signal.
2. **A 10k-step probe with FineWeb-Edu mix demonstrated English emergence.**
   On the same architecture and same compute (`run-v3a-probe`), training
   on a FineWeb mix produced grammatical English continuations (`"Once upon
   a time, in a small village, the city was in a very small area. The city
   was built in the early 18th century..."`). The docs-only baseline at
   matched compute produced disconnected document fragments
   (`"- - name - file = foo to create a path..."`). Greedy decoding
   surfaced mode-collapse loops, but English fluency clearly emerged.

Together these findings argue for re-opening v3 with a different data mix
and a larger SFT corpus. This is the work plan for that follow-up.

## Non-goals (emphasized)

- **Not aiming to match Qwen quality.** Realistic ceiling at 57M params,
  ~520M training tokens, single-Mac compute is roughly 25–35/100 vs Qwen's
  ~80/100 on grounded RAG tasks.
- **Not chasing pretraining-quality benchmarks.** No HellaSwag, no MMLU,
  no MT-Bench. Smoke samples and the existing 50-query diagnostic carry
  the comparison.
- **No cloud GPU spend.** Whole plan runs on the user's M1 MacBook.
- **No HuggingFace pretrained tokenizer.** Self-trained 32k BPE on the new
  mix (re-using `tokenizer_train.py` infrastructure with an incremental
  optimization). Using a pretrained tokenizer would skip a core learning
  goal of the v3 track.

## Subtasks

### 1. Self-trained BPE on FineWeb mix (~6h)

The current 32k BPE tokenizer (`data/tokenizer/tokenizer.json`) was trained
on a 7 MB Python docs corpus and fragments badly on FineWeb text. Retrain
the BPE on the new mix to recover sensible tokenization, while keeping the
self-implemented training pipeline intact (no HF tokenizers library).

**§1.1 — Sample script** (`scripts/sample_corpus.py`): reservoir sampling
to extract a 100 MB representative sample from the 2.4 GB mix corpus.
Bytes-target driven, single-pass streaming.

**§1.2 — Incremental BPE training**: replace the naive
O(unique_words × vocab_size) `train_bpe` with an incremental algorithm.
Key idea: maintain `pair_freqs` and `pair_to_words` indexes across merge
iterations; each merge updates only the words containing the merged pair.
Expected speedup 10–30× (standard BPE optimization documented in
sentencepiece, fastBPE). Keep the naive algorithm as a baseline so a
correctness test (`naive_vs_incremental` on a 100 KB corpus) verifies
output equivalence before the long run.

**§1.3 — Multiprocessing encoder**: extend `tokenizer.encode` with a
parallel batch path (`encode_batch_parallel(texts, n_workers=4)`). Each
worker holds its own per-word cache. Expected ~3× speedup for the full
2.4 GB encode pass.

**§1.4 — Run training**: 30–60 min training on the 100 MB sample; output
written to `data/tokenizer/tokenizer-mix.json`. Vocab size 32k; param
budget unchanged from v3 §4 (57M total model).

### 2. FineWeb-Edu corpus build (~3.5h)

Stream `HuggingFaceFW/fineweb-edu` (subset `sample-10BT`) until ~2.4 GB
of text is collected (≈ 400–500 M tokens after BPE). Mix with the v3
docs corpus tiled to ~5 % bytes (docs is only ~7 MB; tile 17–18× to reach
120 MB).

- New script: `scripts/build_fineweb_corpus.py`
- Output: `data/pretrain/mix_corpus.jsonl` (one JSON per line,
  `{"text": "..."}` schema matching `data/pretrain/corpus.jsonl`)
- Manifest fields (mandatory for reproducibility):
  - FineWeb HF dataset commit hash
  - Mix seed, docs tile count, byte counts (FineWeb / docs)
  - Build timestamp

**Critical:** before tokenizer training, **set aside 5 % of the eventual
segments as a held-out validation set** (separate file or seed-fixed
split). The 200k-step pretrain run is too long to be flown blind without
val loss tracking.

### 3. Encode mix corpus (~1–1.5h run)

Re-use `scripts/encode_corpus.py` with the new tokenizer-mix.json.
Multi-process encode (§1.3 deliverable) brings 2.4 GB → segments tensor
in ~1–1.5h on M1.

Output payload: `{"segments", "seq_len", "vocab_size", "tokenizer_path",
"corpus_path", "build_timestamp", "val_indices"}` (val_indices keys the
held-out 5 %).

### 4. SDPA in attention forward (optional, ~30 min code)

Replace the manual `q @ k.T / sqrt(d); softmax; @ v` block in
`Attention.forward` with `nn.functional.scaled_dot_product_attention`:

- Prefill (`cache=None`): `is_causal=True` to take Metal Flash Attention
  fast path on MPS.
- Decode (cache provided): explicit `attn_mask` derived from
  `~causal_mask[position:position+T, :T_kv]` (SDPA convention is
  `True = attend`).

Empirical: ~17 % training speedup on M1 (probe went from 1.82 step/s to
2.13 step/s). Numerically equivalent to the manual implementation;
existing model tests pass unchanged. **Recommend gating behind a config
flag** (`attention_impl: "manual" | "sdpa"`) so the v3 §1–§8 narrative
(which references the manual implementation as the v3 baseline) stays
faithful.

Re-use `scripts/pretrain_tinydocs.py` once segments has the new

- `--warmup-steps 1000`
- `--checkpoint-every 20000` (~10 ckpts × 600 MB = 6 GB)
- `--log-every 200`
- **`--val-segments PATH` (new)**: compute val loss every 1000 steps,
  log under `val_loss` field
- **`--sample-every 20000` (new)**: generate 5 fixed-prompt samples per
  milestone, write to `samples.jsonl`. Lets us spot mode-collapse trends
  and abort early without waiting 17 h.
- Verbose console logging (rolling step rate, ETA, milestone block,
  finish clock-time).

**Expected loss trajectory** (extrapolated from probe):

| Step | Train loss | Val loss (typical gap) |
|---|---|---|
| 0 | 10.37 (= log 32k) | 10.37 |
| 1k | ~7.0 | ~7.0 |
| 10k | ~4.3 | ~4.4 |
| 50k | ~3.3 | ~3.4 |
| 200k | ~2.6–3.0 | ~2.8–3.2 |

Mode collapse on greedy decode is expected throughout but should reduce
markedly past step 50k.

**Bail-out criteria** (built into the §A.7 narrative as decision points):

| Trigger | Action |
|---|---|
| step 5k loss > 8 | abort — config bug, check LR / init |
| step 50k val_loss > train_loss × 1.2 | overfit early, document and proceed |
| step 100k val_loss rising 5 records in a row | save current ckpt as final, stop |
| chip temp > 90 °C sustained 30 min | force stop, thermal throttling will skew metrics |

### 6. SFT distill from Qwen (~3h compute) ✅ done 2026-05-05

**Actual outcome**: ran a 4-step scaling experiment instead of one 1000-2000
sample run. Best ckpt at 332 effective records; **citation rate 60 %**
(beat plan's 30-50 % expectation). Adding 150 more samples regressed
quality to 20 % — surprise data-scaling plateau at 67M params, documented
in `experiments/v3.1-fineweb-sft-narrative.md`.

| Run | Source | Effective records | Citation rate (5-query smoke) |
|---|---|---:|---:|
| v1 | Qwen 265 | 182 | 60 % |
| v2 | + Claude rejected-redo 50 | 232 | 60 % |
| **v3** ⭐ | + Claude chunks 100 | **332** | **60 %** |
| v4 | + Claude chunks 150 | 482 | 20 % (regression) |

Final SFT ckpt: `data/checkpoints/run-sft-v31-v3/step_final.pt`.

OOS refusal training (24 `[INSUFFICIENT-CONTEXT]` samples in v2) did
**not** transfer — 0/6 refusals on OOS smoke. Refusal needs ≥ 25 % of
the SFT corpus, not 10 %.

Extends the SFT distillation pattern from the feasibility exploration:

- **Larger corpus.** Generate 1000–2000 `(query, retrieved_chunks,
  qwen_answer)` triples. Sources:
  - All three eval sets (`v0_core.jsonl`, `v1_out_of_scope_20.jsonl`,
    `v2_full.jsonl`) deduped by query string (~165 unique)
  - **Synthetic queries from corpus chunks**: for every chunk, ask Qwen
    "generate one user question about this API: {chunk_id} {body}" → adds
    several hundred to a few thousand additional queries grounded in real
    chunks.
- **Cap answers at 128 tokens** in the Qwen sampling step. The 122-sample
  diagnostic lost 38 % of training samples to the 256 max_seq_len truncation
  budget; capping the answer at generation prevents this.
- **SFT data quality gate**: sample 20 random `(query, chunks, answer)`
  triples after generation, manually inspect for grammaticality, citation
  presence (`[N]`), and absence of fabricated APIs. If pass rate < 80 %,
  iterate the grounded prompt.
- **Train hyperparams (revised from feasibility lesson):** lr 5e-6
  × 5–8 epochs (vs the probe's 1e-5 × 3, which produced 18 % mode collapse).
  Loss masked to answer tokens only.

**Expected post-SFT outcomes:**

- `[N]` citation appearance: 30–50 % of outputs (vs near-zero on the
  docs-only base + SFT).
- `[INSUFFICIENT-CONTEXT]` refusal: still rare (Qwen rarely refuses;
  consider injecting 5–10 % synthetic refusals in the SFT corpus).
- LLM-judge "correct" rate vs Qwen 80 %: roughly 25–35 %.

## Risk register

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Incremental BPE diverges from naive | Med | Vocabulary corruption → all training wasted | Required unit test: 100 KB corpus, naive vs incremental must produce identical (vocab, merges); gate further work on this |
| FineWeb HF download stalls | Med | Corpus build fails | Streaming retry + resume; recommend setting `HF_TOKEN` for higher rate limits |
| M1 thermal throttling on 17 h run | High | Step rate drops 30 %, wall-clock +40 % | Plug in mains power + cooling pad; budget ETA with +50 % buffer |
| 200k-step model still mode-collapses on greedy | Med | Smoke output looks bad | §7 evaluates sampled-decode column; if sampling produces fluent output, accept the ckpt |
| Qwen-distilled SFT data has hallucinations | High | Model learns wrong APIs | §6 quality gate; manually audit 20 random samples |
| Disk full mid-run (segments 4 GB + ckpts 6 GB + SFT data) | Low | Run aborts | Verify 30 GB+ free before §5 launch |
| Val loss computation bug (mask, reduction errors) | Med | Misleading early-stop decisions | Sanity check after first val loss compute: should equal train loss at step 0 and trend with train loss thereafter |

## Reproducibility manifest

Every produced artifact (`tokenizer-mix.json`, `mix_corpus.jsonl`,
`segments-mix-256.pt`, every checkpoint) ships with a `manifest.json`
containing:

- **Data**: FineWeb HF dataset commit hash, mix seed, sample seed, val
  split seed, byte counts.
- **Code**: `git rev-parse HEAD`, Python / torch / transformers versions.
- **Hardware**: M1 chip variant, RAM size, macOS version.
- **Training**: timestamps, total steps, total wall-clock.

This is mandatory for the experiment to be replayable later.

## Decision points

- **Tokenizer**: self-trained 32k BPE (recommended — preserves v3 learning
  goal, keeps 57M model) vs HF GPT-2 (faster but skips a core learning
  task). **Choose self-trained.**
- **Pretrain steps**: 100k vs 200k. 200k recommended — at 100k loss is
  ~3.6, at 200k ~2.8, and the difference is visible in coherence and
  mode-collapse rate.
- **SFT corpus mix**: Qwen-only (recommended, free) vs Qwen + GPT-4
  (better quality but $cost).
- **Encoder workers**: 4 (M1 has 4 performance cores; more causes shared
  memory contention).
- **Decoding modes evaluated**: both greedy and sampled (recommended,
  see §7).

## Out-of-scope

- **Cloud GPU.** $50–200 + data movement; stay local.
- **Larger model (200M+).** Needs cloud + full data prep redo; first
  finish polishing the 57M MVP.
- **DPO / RLHF.** Out-of-budget alignment work; grounded RAG tasks already
  cover the citation + refusal alignment surface.
- **Direct Q&A pretraining on synthetic SFT data.** Llama-style "instruct
  base" trick; cleaner to keep base + SFT separated.
- **HuggingFace tokenizers library / pretrained tokenizer.** v3 learning
  goal is self-implemented BPE.
- **Loss tracking on tensorboard.** JSONL logs are sufficient at this
  scale; tensorboard is overhead for a 200k-step single run.

## Success criteria

The plan succeeds iff:

1. The §A base produces fluent English on the 5-prompt smoke (Q2-style
   real sentences, structured code-shaped continuations).
2. §A + SFT produces grounded answers with `[N]` citations on at least
   30 % of identifier-type queries from `v0_core.jsonl`.
3. The 50-query diagnostic (greedy + sampled columns) shows §A+SFT
   superior to §4+SFT on at least 60 % of queries by qualitative side-by-
   side review.
4. Total compute ≤ 30 h M1 pretrain + 3 h M1 SFT, **no cloud spend**.
5. Reproducibility manifest is shipped with every artifact.

The plan **does not need to**:

- Match Qwen quality (impossible at 57M)
- Beat Qwen on any metric (impossible at 57M)
- Pass formal LLM-as-judge eval at high agreement
- Ship anywhere

This is a learning artifact bounded by parameter count.
